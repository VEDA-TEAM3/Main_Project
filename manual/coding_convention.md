# C/C++ 코딩 컨벤션 가이드

---

## 1️⃣ 기본 포맷팅

모든 코드는 프로젝트 루트에 제공된 `.clang-format` 파일에 의해 자동으로 형태가 교정됩니다.

* **들여쓰기 (Indentation):** 스페이스 4칸
* **줄 길이 (Line Length):** 최대 120자 제한
* **중괄호 (Braces):** K&R 변형 스타일 적용
  * 제어문(`if`, `for`, `while`, `switch`)의 여는 중괄호는 같은 줄에 작성합니다.
* **포인터/참조자 위치:** 자료형 쪽에 붙여서 작성합니다. (예: `int* p`, `const Type& ref`)

---

## 2️⃣ 명명 규칙

C와 C++의 패러다임 차이를 고려하여 명명 규칙을 엄격히 분리합니다. 이 규칙은 `.clang-tidy` 정적 분석을 통해 강제됩니다.

| 대상 | C++ | C |
| :--- | :--- | :--- |
| **변수 (Variables)** | `camelCase` | `snake_case` |
| **함수 (Functions)** | `camelCase` | `snake_case` |
| **클래스 (Classes)** | `PascalCase` | (해당 없음) |
| **구조체 (Structs)** | `PascalCase` | `Capitalized_snake_case` (첫 글자 대문자) |
| **매크로 (Macros)** | `UPPER_CASE_WITH_UNDERSCORES` | `UPPER_CASE_WITH_UNDERSCORES` |

## 3️⃣ 주요 코딩 원칙

#### 1. 확장자 및 파일 구성
* **파일 확장자 고정:** 소스 코드는 `.c`, `.cpp`, 헤더 파일은 `.h` 만을 사용합니다. (`.hpp`, `.cc`, `.cxx` 등 혼용 금지)
* **Include Guard 고정:** 모든 헤더 파일의 최상단에는 중복 포함 방지를 위해 `#pragma once`만을 사용합니다.
* **헤더 Include 순서:** 의존성을 낮추기 위해 **시스템 헤더를 최우선**으로 배치합니다.
  1. C 표준 시스템 헤더 (`<stdio.h>` 등)
  2. C++ 표준 시스템 헤더 (`<iostream>` 등)
  3. 외부 라이브러리 및 프로젝트 내부 헤더 (`"my_header.h"`)

#### 2. 상수와 매크로
* **C++:** `#define` 매크로를 이용한 상수 정의를 **전면 금지**합니다. 모든 상수는 `constexpr` 또는 `const`로 선언하여 타입 안정성을 확보해야 합니다.
* **C:** 제한적으로 `#define` 매크로 사용을 허용합니다.

#### 3. 주석 (Comments)
* Doxygen 스타일(`/** ... */`)을 사용합니다.
* 무분별한 주석을 지양하고, **클래스와 함수 단위**에만 집중적으로 Doxygen 주석을 작성하여 API 문서화의 역할을 하도록 합니다.

#### 4. 메모리 관리 (C++ 전용)
* `new` / `delete`, `malloc` / `free`의 직접 호출을 전면 금지합니다.
* 동적 할당이 필요한 경우 **`std::shared_ptr` 사용을 고정 원칙**으로 합니다. (특수한 소유권 독점 상황이 입증된 경우에만 `std::unique_ptr` 허용)

#### 5. 에러 처리 규칙 (C언어 전용)
* **Linux 커널 스타일 적용:** 
  * 함수의 성공 여부를 반환할 때, **성공 시 `0`**, **실패 시 `음수 에러 코드(-EFAULT 등)`**를 반환합니다.
  * 단, 반환값이 `boolean` 성격을 띄는 상태 확인 함수(이름이 `is_` 또는 `has_`로 시작하는 함수)에 한하여 참(성공)일 때 `1`, 거짓일 때 `0`을 반환합니다.
* *(주의: 이 규칙은 Linter가 완벽히 잡아낼 수 없으므로 코드 리뷰 시 반드시 교차 검증해야 합니다.)*
